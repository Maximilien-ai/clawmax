Date: April 13, 2026
Owner: main branch release prep
Current shipped baseline: `v1.2.18`
Current unreleased follow-through on `main`: `03a4cfd`

## Why v1.3.0 Instead of Another Patch

`1.3.0` is the right next cut if this is the version we want initial external users to test.

Reason:
- the shipped surface changed materially across hosted vs local/operator-managed runtimes
- GitHub now has a hosted runtime token path in addition to the local/on-prem `gh` path
- Ollama behavior is now deployment-aware instead of assuming one runtime shape
- onboarding, BYOK, Doctor, prereqs, and partner surfaces all changed in user-visible ways
- partner-specific Blaxel/Redis work was intentionally removed from `main` and isolated to branches

This is no longer just a sequence of isolated point fixes.

## Proposed v1.3.0 Scope

Include:
- hosted GitHub runtime token support and readiness behavior
- local/on-prem GitHub CLI auth flow preservation
- onboarding/BYOK stability fixes
- BYOK save/close consistency and readiness-state fixes
- deployment-aware Ollama UI visibility and default-base-url behavior
- Doctor/runtime messaging cleanup for managed deployments
- stricter `main` partner surface with only `GitHub`, `Senso`, and `Opik`
- prereq fixes for preferred model and packaged skills like `workspace-ls`
- removal of unstable Blaxel/Redis partner/template surfaces from shipped `main`

Do not include:
- unfinished Blaxel-specific runtime/template work
- unfinished Redis-specific runtime/template work
- partner-branch demo helpers
- speculative runtime packaging changes that have not been validated

## Release Blockers Before Tagging 1.3.0

Must be green:
- hosted login on empty workspace
- onboarding stays visible until the workspace is actually initialized
- BYOK save works on first try and readiness state updates immediately
- hosted GitHub token save path works end-to-end
- hosted GitHub readiness reflects token mode, not CLI mode
- `ClawMax System Test` passes in a hosted deployment with GitHub configured
- Doctor reflects the real gateway/runtime contract and stays green when the runtime is healthy
- local/on-prem GitHub CLI auth still works
- local/native Ollama still defaults to `http://localhost:11434`
- containerized on-prem Ollama uses injected `OLLAMA_BASE_URL` without reverting to localhost

Should be green:
- template apply prereqs no longer show false `preferred model` or `workspace-ls` warnings
- `Keys & Secrets` hides `OLLAMA_BASE_URL` when Ollama is disabled
- README and env docs stay generic and do not expose product-specific deployment details

## Tester Checklist

### Hosted / Non-Interactive Deployment

- login to an empty workspace
- confirm onboarding remains visible
- open BYOK / Partners
- save an OpenAI key once
- confirm modal closes on first save
- confirm readiness pill updates
- set a preferred model
- apply `ClawMax System Test`
- confirm prereq warnings about shared execution path and `workspace-ls` are gone
- open `Partners -> GitHub`
- save a runtime token plus default repo
- reopen and confirm:
  - token field is blank
  - UI shows token is configured on server
  - readiness reflects token mode
- rerun `Test GitHub`
- confirm Doctor remains healthy and gateway is green

### Local / Native Dev

- confirm GitHub still offers the CLI auth path
- confirm `gh auth login` path remains functional
- confirm Ollama still defaults to `http://localhost:11434`
- confirm local model listing still works

### Containerized On-Prem

- set:
  - `DASHBOARD_ENABLE_OLLAMA=true`
  - `OLLAMA_BASE_URL=http://host.containers.internal:11434`
- confirm Ollama is visible in the dashboard
- confirm BYOK shows the injected URL instead of localhost
- confirm model listing works from inside the containerized deployment

## Documentation Needed Before Tagging

## Work That Can Happen Before the Tag

Safe to do:
- documentation cleanup
- release-candidate checklist tightening
- template canon audit planning
- backlog cleanup

Do not do blindly:
- more hosted runtime behavior changes without a fresh image/test cycle
- more gateway process-management changes without the CLI/runtime details
- more partner-surface expansion on `main`

## Recommended Cut Strategy

1. Treat the next tag as `v1.3.0-rc1` internally, even if the GitHub release is just a normal release candidate workflow.
2. Roll the current unreleased Ollama follow-through into that candidate.
3. Run the hosted, local/native, and containerized on-prem checks above.
4. If green, tag `v1.3.0`.
5. If a small hosted-only regression appears, fix it once and retest instead of continuing the `1.2.x` patch line.
